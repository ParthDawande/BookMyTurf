PHASE 9-customer-lifecycle-1 — My Bookings list + full Receipt page
+ Cancel flow.

Standing rules apply. Stop and report. Do NOT start
9-customer-lifecycle-2 (reschedule + reviews). Reuse Phase 0 error
shape, Phase 2 security UNCHANGED. The backend is frozen — build
against existing endpoints. No new dependencies.

SCOPE: three customer-facing features around viewing and cancelling
their bookings.
- /customer/bookings — paginated list of customer's own bookings.
- /customer/bookings/:id — full receipt page (replaces 9-customer-
  booking's placeholder).
- Cancel flow — confirmation modal from the receipt, calls DELETE,
  page reflects new state.

=== Locked design decisions ===
- P1: cancel button is ALWAYS VISIBLE on CONFIRMED bookings, but
  DISABLED with an inline explanation when outside the 24h window.
  Same approach will apply to reschedule button in lifecycle-2.
- P5: page state update after cancel IS the customer's confirmation.
  No customer-facing notification, no toast on top of the page
  update. The page-state-change is the confirmation.

=== Backend endpoints used (existing, do not modify) ===
- GET /api/customer/bookings (paginated list, snake_case query
  params per build convention: ?page=&size=&status=).
- GET /api/customer/bookings/{id} (full receipt — uses the 6A
  receipt-fix shape with payments PLURAL array, refunds array).
- DELETE /api/customer/bookings/{id} (cancel-with-refund per 6B).

State the exact response shapes you observe from each (the
fields you depend on), since this informs the frontend's rendering.

=== Page 1: My Bookings list (/customer/bookings) ===

Layout:
- Header (existing Header.jsx with user email + logout).
- Page title: "My Bookings."
- Status filter: dropdown with options "All", "CONFIRMED",
  "COMPLETED", "CANCELLED", "REFUNDED". Default "All".
  Filter changes the ?status= URL param and refetches.
- Booking list: card per booking with:
  - Turf name (clickable → /turfs/{turf_id} for detail).
  - Sub-court name + sport.
  - Booking date (human-readable: "Saturday, June 27, 2026").
  - Slot time range (first slot start → last slot end:
    "9:00 AM – 11:00 AM").
  - Status badge with visual treatment per status (color-coded:
    CONFIRMED green, COMPLETED neutral, CANCELLED grey-strike,
    REFUNDED amber/red — use the design tokens).
  - Total amount.
  - "View details" CTA → /customer/bookings/{id}.
- Pagination controls at the bottom.
- "Book another" CTA somewhere visible → /turfs.

States:
- Loading: simple "Loading..." or spinner using the existing pattern.
- Empty (no bookings): "You haven't made any bookings yet. Browse
  turfs →" with the browse CTA. Required for v1.
- Error: error message via the existing ErrorBanner/useApiError
  pattern.

URL semantics:
- /customer/bookings?status=CONFIRMED&page=0 — bookmarkable.
- Status="All" doesn't include the param (cleaner URL).

=== Page 2: Receipt page (/customer/bookings/:id) ===

This replaces the placeholder from 9-customer-booking. The receipt
is the customer's record of the booking, and it has to honour the
multi-payment/multi-refund shape from the 6A receipt fix.

Layout:
- Header with status badge prominently displayed (same color/
  treatment as the list).
- Turf, sub-court (clickable → /turfs/{turf_id}).
- Date (human-readable).
- Slots: list each slot's time range. For a 2-slot booking show
  both slot ranges ("9:00 AM – 10:00 AM", "10:00 AM – 11:00 AM").
- Total amount currently owed (bookings.total_amount).
- PAYMENTS section: render the payments array.
  - For a normal booking (1 payment): "Paid ₹X.XX on [date]".
  - For a multi-payment booking (PAYMENT-reschedule history):
    list each row chronologically: "Original payment: ₹X.XX on
    [date]", "Additional charge: ₹X.XX on [date]". Display the
    payment_method opaque string as-is ("card", "netbanking",
    "upi" — don't transform; trust the wire format).
- REFUNDS section (only if refunds array is non-empty):
  - For a cancelled booking (full refund): "Refunded ₹X.XX on
    [date]".
  - For a reschedule-REFUND (partial refund): "Reschedule
    refund: ₹X.XX on [date]".
  - Each entry shows amount and date. The razorpay_refund_id is
    optional (small text, or hidden — your call).
- Action buttons (depend on booking status — see below).

Action button logic:
- Status CONFIRMED AND earliest slot >24h in the future:
  Cancel button ENABLED.
  Reschedule button ENABLED (placeholder in this sub-phase;
  9-customer-lifecycle-2 implements the flow). On click in
  this sub-phase show "Reschedule coming in next sub-phase."
- Status CONFIRMED AND earliest slot ≤24h or in past:
  Cancel button VISIBLE but DISABLED. Reschedule same.
  Inline text below the buttons: "Cancel and reschedule are
  unavailable within 24 hours of start time."
- Status COMPLETED: no cancel/reschedule buttons. "Leave a
  review" button placeholder (9-customer-lifecycle-2). Show
  "Reviews coming in next sub-phase" on click.
- Status CANCELLED or REFUNDED: no action buttons. The status
  badge tells the story.

Date comparison for the 24h window:
- The backend enforces this with Asia/Kolkata→Instant comparison.
- The frontend can compute the same client-side using
  toZonedTime or by treating the booking_date + slot start_time
  string as IST, converting to a Date via Date('YYYY-MM-DDTHH:mm:00+05:30'),
  and comparing to new Date(). State your approach.
- A small precision detail: the frontend's "24h" check determines
  whether the BUTTON is enabled. The backend's gate is authoritative
  — if the frontend allows a click (button enabled) and the backend
  rejects (just-crossed-the-window edge), the error mapping should
  surface it cleanly. State that you don't try to second-guess the
  backend.

States: same loading/error/not-found patterns as elsewhere. A 404
from the backend (booking-not-yours or non-existent) shows "Booking
not found" with a back-to-list CTA.

=== Cancel flow ===

Triggered by clicking the enabled Cancel button on the receipt
page. Opens a confirmation modal:
- Title: "Cancel this booking?"
- Body: "You'll be refunded ₹X.XX to your original payment method.
  This cannot be undone."
- "Cancel booking" primary button (destructive treatment — red).
- "Keep booking" secondary button (closes modal).

On "Cancel booking" confirm:
- DELETE /api/customer/bookings/{id}.
- 200 OK: page state updates IMMEDIATELY. Status badge changes to
  REFUNDED. Refund row appears in the refunds array (the response
  contains the updated booking state, or refetch). Cancel/
  reschedule buttons disappear. NO toast, no notification — the
  page change is the confirmation.
- 400 (e.g., not in 24h window — shouldn't happen if button was
  enabled correctly, but possible at the edge): error banner with
  the backend's message.
- 502 (refund-failure path from 6B-ii): error banner "Cancellation
  failed. Your booking is unchanged. Please try again or contact
  support." with a "Contact support" link (placeholder). DO NOT
  imply the cancellation succeeded; the booking stays CONFIRMED
  and the FAILED refund row is invisible to the customer for v1
  (admin-queue concern).
- Network error: "Couldn't reach the server. Please check your
  connection and try again."

=== Carried-forward contracts to enforce in this sub-phase ===

1. payment_method is opaque varchar. Display whatever the wire
   format returned. Don't transform "card" to "Credit Card" or
   any cosmetic remap unless the backend returns the cosmetic
   form already.
2. The receipt's payments array can have 1+ entries. For booking
   33 (from 9-customer-booking) it has 1. For owner 20's
   booking 21 (from 6C-iii-1b) it has 2. Both render correctly.
3. The receipt's refunds array can have 0+ entries with various
   sources (cancel, reschedule, race-recovery — but race-recovery
   is FILTERED OUT by the 6A endpoint, so the customer never sees
   booking_id IS NULL rows). Trust the backend's filter.
4. P1: cancel button disabled with inline text when outside the
   24h window.
5. P5: no customer-facing notification; page state is the
   confirmation.

- No reviews (lifecycle-2).
- No backend changes.
- No notifications-list page (already exists from 8C; this
  sub-phase doesn't touch it).
- No new dependencies.
