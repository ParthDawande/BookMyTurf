the relevant fields per the backend's CreateOwnerRequest DTO. Likely
fields (verify via DTO):
- Name (required, non-empty).
- Email (required, valid format).
- Phone (required — Indian phone, 10 digits, per Phase 2's pattern).
- Password (required, with backend's complexity rules).
- City (required).

register by default; owners can switch from there or visit the

=== /owner/dashboard page ===

Layout:
- Header (existing) + OwnerNav.
- Page title: "Dashboard."
- Date range filter at the top: two date inputs (from / to) plus
  a "Clear filters" button. Optionally a turf filter dropdown
  listing the owner's turfs (call the existing /api/owner/turfs
  endpoint if it exists, or skip the turf filter for v1 if it
  doesn't — state your choice). The 8B-owner endpoint supports
  turf_id but if a turf dropdown is complicated for v1, you can
  defer the dropdown to 9-owner-turf-management and just support
  the date filter here.

Main content — KPI cards layout (4-column on desktop, stacking
on mobile):

Row 1 — Revenue cards:
- Total Revenue: ₹X (large number).
- Platform Commission: ₹X (smaller, secondary treatment —
  visible to owner for transparency, but not the headline).
- Your Earnings: ₹X (the owner_payout_total, equally prominent
  as Total Revenue).

Row 2 — Bookings by status (a small grouped section):
- CONFIRMED: N (green).
- COMPLETED: N (neutral).
- CANCELLED: N (grey).
- REFUNDED: N (amber/red).

Row 3 — Payouts summary cards:
- Pending Payout: ₹X (amber).
- Paid Out: ₹X (green).
- Cancelled Payouts: ₹X (grey).

States:
- Loading: skeleton or simple loading text for each card.
- Error: existing ErrorBanner.
- Empty state (a brand-new owner with no data): all zeros
  rendered cleanly, with a "Add your first turf" CTA → links
  to /owner/turfs (which is a stub but the link is the right
  forward shape).

Date filter behavior:
- ?from_date=YYYY-MM-DD&to_date=YYYY-MM-DD applied as URL state.
- Submitting refetches the dashboard with the filter.
- Snake_case URL params (carried-forward rule).
- The carried-forward dual-domain rule from 8B-owner applies: the
  same range filters both booking-domain numbers (revenue,
  bookings_by_status) and payout-domain numbers (payouts_summary)
  by their respective fields. Don't try to second-guess that —
  the backend handles it.

=== Carried-forward contracts to enforce ===

1. The 8B-owner response shape: total_revenue is for CONFIRMED +
   COMPLETED only; REFUNDED/CANCELLED counts appear in
   bookings_by_status but contribute 0 to revenue. Display the
   bookings_by_status counts AS-RECEIVED — don't filter.
2. Snake_case end-to-end.
3. owner_payout_total = total_revenue - platform_commission. The
   backend guarantees this is internally consistent (verified in
   8B-owner). Frontend just displays the field directly; don't
   re-compute client-side.
4. The reusable NotificationsPage component should now be wrapped
   in OwnerNav when mounted at /owner/notifications.
