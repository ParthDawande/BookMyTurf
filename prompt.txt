SCOPE: complaints lifecycle — customer raises, admin assigns to staff,
staff (or admin) resolves. complaint_notes writes by staff/admin. One
owner-directed-equivalent notification on assignment: COMPLAINT_ASSIGNED
to the staff member. Endpoint paths per API_DOC.md.

=== Locked design decisions (from Phase 7 framing) ===
- Q1: complaint_notes are INTERNAL-ONLY by default. Customer's detail
  view of their own complaint does NOT include the notes array. Staff
  and admin views DO include notes.
- Q2: admin can reassign a complaint at any time before resolution.
  Reassign changes assigned_staff_id, emits a new COMPLAINT_ASSIGNED
  notification to the NEW staff. No "unassigned" notification to the
  old staff (§5 doesn't define such a type).
- Q3: resolution REQUIRES a non-empty resolution_note. The resolve
  endpoint accepts a resolution_note field; the service writes it as
  a complaint_notes row, then flips status to RESOLVED in the same
  transaction.
- Q4: admins can override-resolve a complaint (even if not assigned).
  Non-assigned staff attempting to resolve → 404 (no-leak).
- Q5: complaint_notes XOR semantics — every row this sub-phase writes
  must have complaint_id SET and query_id NULL. The XOR is enforced
  application-side (the schema does not enforce it). 7C will write
  the inverse.

=== Status flow (DECISIONS §3-style semantics) ===
OPEN → IN_PROGRESS (admin assigns)
IN_PROGRESS → RESOLVED (assigned staff or admin)
No other transitions. OPEN cannot become RESOLVED directly (must be
assigned first). RESOLVED is terminal. Status guard violations → 400
with clear messages (state-transition violations use 400 per
DECISIONS §4 — not 422, not 409; 409 is for resource uniqueness).

=== Endpoints (per API_DOC paths) ===

CUSTOMER:
- POST /api/customer/complaints — create. Body: subject, description,
  related booking_id (optional). Validation: subject and description
  non-empty. If booking_id given, must be owned by the customer.
- GET /api/customer/complaints — list customer's own complaints,
  paginated per API_DOC. Status visible. Notes NOT included.
- GET /api/customer/complaints/{id} — detail. Customer can only see
  their own (404 otherwise, no-leak). Notes NOT included.

ADMIN:
- GET /api/admin/complaints?status= — list, filterable by status,
  default OPEN. Notes included.
- GET /api/admin/complaints/{id} — detail with notes array.
- POST /api/admin/complaints/{id}/assign — body: staff_id. Validates
  that staff_id is a STAFF role user. Status OPEN → IN_PROGRESS,
  assigned_staff_id set. Fires COMPLAINT_ASSIGNED notification to
  the staff. If complaint already IN_PROGRESS, this is a reassign:
  change assigned_staff_id, fire new COMPLAINT_ASSIGNED to the new
  staff, no notification to old staff. Status stays IN_PROGRESS.
  RESOLVED complaints cannot be (re)assigned → 400.
- POST /api/admin/complaints/{id}/resolve — body: resolution_note
  (non-empty required). Admin can resolve any complaint, regardless
  of who is assigned. Same behavior as the staff resolve below for
  the DB writes.

STAFF:
- GET /api/staff/complaints — list complaints assigned to this staff
  (assigned_staff_id == caller). Notes included.
- GET /api/staff/complaints/{id} — detail. Only visible if assigned
  to this staff (404 otherwise, no-leak).
- POST /api/staff/complaints/{id}/notes — body: note (non-empty
  required). Writes a complaint_notes row: complaint_id = this
  complaint, query_id NULL, author_id = caller, note_text, created_at.
  Only the assigned staff (or an admin via the admin endpoint, see
  below) can add notes. Returns the new note.
- POST /api/staff/complaints/{id}/resolve — body: resolution_note
  (non-empty required). Only the assigned staff. Writes a
  complaint_notes row as the resolution note, then flips status to
  RESOLVED. One transaction.

ADMIN (mirror of staff for note operations, since admins have full
access):
- POST /api/admin/complaints/{id}/notes — same as staff notes
  endpoint, but admin can write notes on any complaint regardless of
  assignment. author_id = the admin caller. Optional in this sub-phase
  if API_DOC doesn't specify; state whether you built this. If not
  built, admins can still resolve (which writes a note), but can't
  add intermediate notes without temporarily assigning to themselves.

=== Authorization summary ===
- Customer endpoints: customer role only. Wrong-customer → 404.
- Admin endpoints: admin role only. Other roles → 403.
- Staff endpoints: staff role only. Wrong-staff (not assigned) → 404.
- Cross-role attempts (e.g., admin token on /api/staff/complaints):
  reject 403. The role-scoping from Phase 2 already does this.

=== Notification scope (§5) ===
- ONE notification type from 7B: COMPLAINT_ASSIGNED to the
  assigned staff. Reassign fires another COMPLAINT_ASSIGNED to the
  new staff.
- ZERO customer-facing notifications (§5 — "Resolved" notifications
  to customer are out of v1 scope).
- ZERO notifications to admin (admins poll the list).
- The notification message should reference the complaint and its
  subject. Format per API_DOC if specified.

=== complaint_notes write rules ===
- complaint_id = the complaint (NOT NULL).
- query_id = NULL (XOR constraint — 7C writes the inverse).
- author_id = caller (staff or admin).
- note_text = the note content, non-empty.
- created_at = NOW().
- Notes are append-only in 7B: no edit endpoint, no delete endpoint.
  (Resolution-note becomes immutable history once the complaint is
  resolved.)

=== Status-transition guards ===
- Assign on a complaint that's already RESOLVED → 400.
- Resolve on a complaint that's OPEN (never assigned) → 400, message
  like "Complaint must be assigned before resolution."
- Resolve on a complaint that's already RESOLVED → 400 (idempotent
  rejection, not a 200 no-op — same pattern as Phase 4's already-
  approved turf).
- Notes on a RESOLVED complaint: REJECT 400 (resolution closes the
  complaint; no further notes). State this explicitly.

=== JPA mapping hazard (from 7A) ===
If you set up complaint_notes with a bidirectional @OneToMany from
Complaint with orphanRemoval=true, you risk the same silent-rollback
bug 7A surfaced when deleting a child via the child repo. 7B does
not have a delete-note endpoint, so this may not bite — but state
explicitly the mapping you used and confirm there's no path that
would trigger orphanRemoval silent rollback. If in doubt, use the
@Modifying JPQL pattern from 7A.
