PHASE 9-admin-operations — Approval queues (turfs + sub-courts),
user management with search, staff management, complaints/queries
admin override. The largest single sub-phase of the build.

Standing rules apply. Stop and report. Do NOT start 9-staff. Reuse
Phase 0 error shape, Phase 2 security UNCHANGED. The backend is
frozen for substantive business logic; small surgical additions
allowed per the established pattern if a needed endpoint doesn't
exist (verify first, decide separately). No new dependencies.

SCOPE: four operational feature areas in one bundle. Replaces the
five stubs in /admin/* from 9-admin-foundation-and-dashboard:
- /admin/approvals — pending turfs + pending sub-courts queues
  with approve/reject.
- /admin/users — paginated list with search by email/name +
  role filter, suspend/ban/activate actions.
- /admin/staff — staff list with create/deactivate/reactivate.
- /admin/complaints + /admin/queries — admin-override views
  with reassign (complaints only) and resolve actions.

=== Locked design decisions ===
- Q1: Approve via confirmation modal (no reason required). Reject
  via modal with REQUIRED reason field (admin must type why).
  Backend's Phase 4 reject endpoint probably accepts a reason
  field — verify and use whatever shape the DTO has. If the
  backend doesn't require a reason, the frontend still requires
  it as UX policy and sends it as part of the request.
- Q2: User list paginated + search by email/name + role filter.
  Backend endpoint shape needs verification via curl — likely
  ?role=&search=&page=&size=. State actual shape.
- Q3 (default): Admin sets initial password when creating staff.
  Display a banner on staff list: "New staff should reset their
  password after first login. (Password reset flow is a v2 polish
  item.)"
- Q4 (default): Inline actions on /admin/complaints and
  /admin/queries — list view IS the action surface, with
  Reassign + Resolve buttons on each card. Modals for both
  actions with required fields.
- Confirmation modals for all destructive/material actions (ban,
  reject, deactivate-staff, resolve, reassign). Suspend/activate
  also confirm.

=== /admin/approvals page ===

Layout:
- Header + AdminNav with Approvals active.
- Page title: "Pending Approvals".
- Two sections (tabs or two grouped lists):
  - Turfs awaiting approval (count from dashboard's
    pending_approvals.turfs).
  - Sub-courts awaiting approval (count from
    pending_approvals.sub_courts).

For each PENDING turf:
- Cover photo (or placeholder).
- Turf name, city, owner (name + email).
- Description excerpt.
- Sub-courts list (count or first few).
- Approve button (green) + Reject button (red).

Approve flow:
- Click Approve → confirmation modal: "Approve {turf name}?
  This turf will become visible to customers."
- Confirm → PUT /api/admin/turfs/{id}/approve. 200 → card
  disappears from queue. No toast.
- Error → ErrorBanner.

Reject flow:
- Click Reject → modal with required textarea "Reason for
  rejection (the owner will see this)." Submit disabled if
  textarea is empty.
- Submit → PUT /api/admin/turfs/{id}/reject with
  {reason: "..."}. 200 → card disappears.
- Error → ErrorBanner.

Same flows for sub-courts. Sub-court approval is independent
of turf approval (a turf can be APPROVED while one of its
sub-courts is still PENDING per Phase 4 contract).

Empty state: "No pending approvals." Shown when both queues
empty.

=== /admin/users page ===

Layout:
- Header + AdminNav with Users active.
- Page title: "User Management".
- Filter row:
  - Role dropdown: All / CUSTOMER / OWNER / STAFF / ADMIN.
  - Status dropdown: All / ACTIVE / SUSPENDED / BANNED.
  - Search input: "Search by email or name..." Submits on
    enter or button click.
- User table or card list:
  - Columns: Name, Email, Role, Status (badge color-coded),
    Joined date, Actions.
  - Actions per status:
    - ACTIVE: Suspend button + Ban button.
    - SUSPENDED: Activate button + Ban button.
    - BANNED: Activate button (un-bans).
- Pagination at the bottom.

All actions use confirmation modals:
- "Suspend {user name}? They won't be able to log in until
  reactivated."
- "Ban {user name}? This is a strong action — they won't be
  able to access the platform. You can reactivate later if
  needed."
- "Activate {user name}? They'll regain platform access."

Each modal action → PUT to the appropriate endpoint. List
refetches on success.

URL semantics:
?role=OWNER&status=ACTIVE&search=verify&page=0&size=20 (snake_case).
Bookmarkable.

Empty state: "No users match your filters."

=== /admin/staff page ===

Layout:
- Header + AdminNav with Staff active.
- Page title: "Staff Management".
- Banner at top: "New staff should reset their password after
  first login. (Password reset flow is a v2 polish item.)"
- "+ Add new staff" CTA at the top right.
- Staff table:
  - Columns: Name, Email, Status, Joined, Actions.
  - Actions: Deactivate (if active) / Activate (if not active).

Create staff modal:
- Fields: name (required), email (required, valid), phone
  (required), password (required, with backend's complexity
  rules — show inline hint about rules per Phase 2 spec).
- Submit → POST /api/admin/staff. 201 → modal closes, list
  refetches with new staff visible.

Deactivate/activate flows: confirmation modal → PUT → list
refetches.

States: loading, error, empty.

=== /admin/complaints page ===

Layout:
- Header + AdminNav with Complaints active.
- Page title: "Complaints" (with badge showing open count
  matching dashboard's open_complaints).
- Filter row: Status (All / OPEN / IN_PROGRESS / RESOLVED).
- Complaint table or card list:
  - Customer name + email.
  - Subject + description excerpt.
  - Related booking (if booking_id set, link to admin's
    booking-view — that doesn't exist as a route, so just
    show "Booking #X at Turf Y" as text).
  - Status badge.
  - Assigned to (staff name or "Unassigned").
  - Created date.
  - Actions: Reassign + Resolve (conditional on status).

Reassign flow (only for OPEN or IN_PROGRESS):
- Modal: "Reassign complaint #X to:" with staff dropdown
  (populated from GET /api/admin/staff or whatever staff
  list endpoint is available). Required selection.
- Submit → PUT /api/admin/complaints/{id}/reassign with
  {staff_id}. 200 → list refetches.

Resolve flow (admin override, only for OPEN or IN_PROGRESS):
- Modal: "Resolve complaint #X" with required resolution_note
  textarea.
- Submit → PUT /api/admin/complaints/{id}/resolve with
  {resolution_note}. 200 → list refetches.

Complaint detail view: optional. If the list shows enough info
inline, no detail page needed in v1. Verify the list response
includes all needed fields (description, customer info, etc.).
If not, add a /admin/complaints/{id} detail page.

States: loading, error, empty.

=== /admin/queries page ===

Same shape as complaints but without reassign (per 7C
atomic-claim model — queries don't have static assignment to
reassign).

Filter row: Status (All / OPEN / IN_PROGRESS / RESOLVED).
Each query card: customer, subject, description excerpt,
status badge, picked_up_by_staff (name or "Unclaimed"),
created date, Resolve action.

Resolve flow: same as complaints (modal with required
resolution_note → PUT /api/admin/queries/{id}/resolve).

=== Carried-forward contracts ===

1. P5: page-state-as-confirmation. After approve/reject/
   suspend/ban/resolve/reassign, the list refetches and
   reflects the new state. No toast.
2. Snake_case throughout.
3. 404-not-403 on cross-resource ownership where applicable
   (mostly N/A for admin — admin sees everything).
4. Backend enforces resolution_note non-empty per 7B; frontend
   gates submit if textarea empty.
5. Backend enforces one resolve per complaint per 7B; if admin
   resolves a complaint that's already RESOLVED, 400 from
   backend.
6. Phase 2 routing: per-request DB status check. Admin actions
   on a user that admin just banned (suspend then ban quickly)
   — the per-request check applies on subsequent requests, so
   the user is locked out at their next page load regardless
   of session state.
7. Notifications: §5 says owner gets TURF_APPROVED /
   TURF_REJECTED / SUBCOURT_APPROVED / SUBCOURT_REJECTED when
   admin acts on their turf/sub-court. Staff gets
   COMPLAINT_ASSIGNED when admin reassigns. These notifications
   fire from the backend automatically; the admin doesn't need
   to do anything special in the UI to trigger them. Verify
   they fire as a side effect of admin actions.
