=== LOCKED DESIGN ===
- Sequence A: Razorpay refund of the additional charge FIRST, then
  commit recovery rows via REQUIRES_NEW. If the booking @Transactional
  rolls back (the booking side does nothing on race), the recovery
  rows must survive because they're written on a separate connection
  that committed independently. Same pattern as 5C-2b.
- Q1a: recovery rows' booking_id is NOT NULL (set to this booking).
  This is the deliberate distinction from 5C-2b race-recovery rows
  (which have booking_id IS NULL because no booking exists). Here a
  booking DOES exist and the customer paid against it specifically;
  the recovery rows belong to that booking's audit trail.
- The booking's own state (slots, total, commission, payout,
  notifications, status) MUST remain completely unchanged. The
  booking transaction creates zero booking-side rows AND modifies
  zero existing booking rows. Only the REQUIRES_NEW recovery rows
  are written, and they're new payments + refunds rows, not edits
  to the booking.
- Booking status stays CONFIRMED throughout. The reschedule did
  not happen; the original booking is intact and playable.
- opaque payment_method (Phase 1 Option B carried contract): the
  recovery payments row stores whatever Razorpay returned as the
  method, AS-IS varchar.

=== Race-after-payment flow (replaces 6C-iii-1b's placeholder) ===

After 6C-iii-1a's three rejection paths pass, the outer
@Transactional starts, SELECT FOR UPDATE on the sub-court row
acquires, and the race re-check runs (excluding own slots). If
any proposed new slot is taken by another CONFIRMED/COMPLETED
booking:

1. CALL RAZORPAY REFUND of the additional-charge payment FIRST.
   The payment_id is razorpay_payment_id from the request (the
   customer just paid). Refund the FULL captured amount (which
   equals serverPriceDiff for this case).
   razorpayClient.payments.refund(paymentId, {amount: paise}).
   Wrap in try/catch on RazorpayException + IOException.

2. ON REFUND FAILURE: rethrow as 502 with the standard body
   {"error":"Payment refund could not be processed"}. 6C-iii-2
   does NOT write a FAILED record here — 6C-iii-3 will harden
   that path using the 5C-3/6B-ii pattern. Add code comment:
   "6C-iii-2: race-after-payment refund failure rethrows 502 with
   no FAILED record; 6C-iii-3 writes the FAILED recovery row
   using REQUIRES_NEW."

3. ON REFUND SUCCESS: call the recovery service (extending
   RaceRefundRecoveryService or whatever you named it) to write
   the recovery rows via REQUIRES_NEW. The recovery method:
   - Take parameters: bookingId, razorpayPaymentId, razorpayRefundId,
     amount, paymentMethod (opaque string from Razorpay).
   - Inside the REQUIRES_NEW @Transactional method:
     a. Create a payments row:
        - booking_id = the bookingId (NOT NULL — Q1a).
          State explicitly that this differs from the 5C-2b race
          recovery where booking_id was NULL.
        - razorpay_payment_id = the razorpayPaymentId.
        - gateway_transaction_id = same value (mirror existing
          payments-row convention from 5B/6B/6C-iii-1b).
        - amount = the captured-then-refunded amount.
        - status = SUCCESS (the CAPTURE succeeded; only the
          reschedule didn't go through; this row records the
          captured payment).
        - payment_method = the opaque string Razorpay returned
          (carried contract; no whitelist or rejecting enum).
     b. Create a refunds row:
        - booking_id = the same bookingId (NOT NULL).
        - payment_id = the just-written payments row's id.
        - razorpay_refund_id = the refund id from the Razorpay
          response.
        - amount = same as the payment amount.
        - status = SUCCESS (the refund completed).
     c. Both rows committed independently via REQUIRES_NEW —
        they survive the outer booking @Transactional's rollback.

4. AFTER the REQUIRES_NEW commit returns: throw
   ResponseStatusException(CONFLICT) with body
   {"error":"Selected slot is no longer available"}. The outer
   @Transactional rolls back — the booking side did nothing
   (no slot delete, no slot insert, no booking total change, no
   payout change, no notification). The recovery rows from step 3
   are already committed on a different connection; they survive.

5. The customer sees 409, their money has been returned
   (Razorpay refund succeeded), and the DB has a queryable record
   of "this customer paid X additional toward a reschedule that
   didn't happen; the money was refunded." Reconciliation works:
   SELECT * FROM payments WHERE booking_id IS NOT NULL is the
   booking's payment audit trail; the additional-charge-then-
   refunded entry is visible.
