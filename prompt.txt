SCOPE: the end-to-end booking flow for an authenticated customer.
Three things land in this sub-phase:
- Multi-slot selection on the existing availability grid in
  TurfDetail (with contiguous-only enforcement).
- A review step (page or modal) showing booking details + total,
  with "Pay now" CTA.
- Real Razorpay checkout.js modal integration: load the SDK,
  instantiate with backend's order data, handle success/dismiss
  callbacks, POST confirm to the backend, navigate to receipt
  on success.

=== Locked design decisions ===
- Q1: multi-slot selection. Customer clicks free slots to toggle
  selection; selection is reflected visually (selected slots
  highlighted distinctly from free/taken); a running total
  appears as slots are selected.
- Q1 sub-rule (contiguous-only): selected slots MUST be a
  contiguous block within the same sub-court. v1 enforcement:
  if a customer clicks a non-adjacent slot, EITHER (a) replace
  the current selection with the new slot OR (b) show an inline
  message "Slots must be contiguous; pick adjacent slots."
  Implementation choice (a) is the simpler/cleaner UX and is
  the recommended v1 behavior. State which you implemented.
- Cross-sub-court selection: NOT allowed. Selection is scoped
  to one sub-court at a time. Clicking a slot on a different
  sub-court replaces the selection.
- Q2: review step between slot selection and Razorpay. This can
  be either a separate page (/booking/review with state passed
  via router) or a modal overlay on the detail page. Modal is
  simpler v1; recommend modal.

=== Backend endpoints used (existing, do not modify) ===
- POST /api/customer/bookings/initiate — body includes sub_court_id,
  booking_date (YYYY-MM-DD), slots array (each slot has start_time
  and end_time as HH:MM strings, or whatever shape the existing
  endpoint accepts — match what 5B verified). Response: razorpay_
  order_id, razorpay_key_id, amount_in_paise (or whatever field
  names the backend returns; state the exact shape observed).
- POST /api/customer/bookings/confirm — body: razorpay_order_id,
  razorpay_payment_id, razorpay_signature. Response on success
  includes booking_id and status; on failure returns 409/502/
  400 per the carried-forward error contract.

=== Slot selection UI (in TurfDetail) ===
On the existing availability grid (one sub-court at a time):
- Free slot: green, clickable.
- Taken slot: grey, not clickable, has "Taken" label.
- Selected slot: distinct color (e.g., blue or a brighter green
  with a checkmark), clickable to deselect.
- A persistent summary panel near the grid (or below it on mobile):
  "{n} slot(s) selected: {start}-{end} · ₹{total}" with a "Book
  now" CTA. The CTA is disabled when 0 slots are selected.
- Total = number of selected slots × sub_court.hourly_rate.
  Display as ₹X.XX with two decimals.

Contiguous-only behavior (recommended choice (a)):
- Currently selected slots form a contiguous block (or empty).
- Click a free slot adjacent to the current block → extends the
  block.
- Click a slot that's currently selected → if it's an endpoint
  of the block, shrink the block from that end; if it's interior,
  shrink from the clicked endpoint.
- Click a free slot NOT adjacent to the current block → REPLACE
  the current selection with just that slot.
- Click a slot on a different sub-court → REPLACE the selection.
- State the exact rules you implemented.

=== Review step (modal recommended) ===
On "Book now" click, the modal/page shows:
- Turf name and city.
- Sub-court name and sport.
- Date (formatted human-readable, e.g., "Saturday, June 27, 2026").
- Slots (formatted as a range: "9:00 AM – 11:00 AM" for 2
  contiguous slots, even though the wire format is HH:MM).
- Total amount: ₹X.XX.
- A "Pay ₹X.XX now" primary CTA.
- A "Cancel" secondary CTA that returns to the detail page with
  the selection intact.

Clicking "Pay" calls the booking-initiate flow.

=== Razorpay checkout.js integration ===

1. Loading the SDK:
   - Load https://checkout.razorpay.com/v1/checkout.js once (cache
     in a module-level boolean or React state). Subsequent
     bookings reuse the loaded SDK.
   - State how you handled SDK loading: dynamic <script> append
     in useEffect, or a utility module.

2. Initiate:
   - On "Pay" click, POST /api/customer/bookings/initiate with
     sub_court_id, booking_date (YYYY-MM-DD), and slots payload.
     Match the exact request body shape the backend expects from
     5B; if the body shape isn't documented in your earlier work,
     verify via a single curl test against a known booking before
     building the UI around it. State the request body shape.
   - Capture the response's razorpay_order_id, razorpay_key_id,
     and amount.

3. Instantiate Razorpay:
   - const rzp = new window.Razorpay({
       key: <razorpay_key_id>,
       order_id: <razorpay_order_id>,
       amount: <amount_in_paise>,
       currency: "INR",
       name: "BookMyTurf",
       description: "<turf name> - <sub-court name>",
       prefill: { email: user.email, contact: user.phone_number },
       handler: <success callback>,
       modal: { ondismiss: <dismiss callback> }
     });
   - rzp.open();
   - State the exact options object you passed.

4. Success callback (handler):
   - Receives { razorpay_payment_id, razorpay_order_id,
     razorpay_signature }.
   - POST /api/customer/bookings/confirm with those three values.
   - On 200 → navigate to /customer/bookings/{booking_id} (the
     receipt route — placeholder is fine for this sub-phase if
     the receipt page proper is in the next sub-phase, but the
     navigation target must be the booking_id).
   - On 409 race auto-refund → show "This slot was just taken.
     Your payment has been refunded — please try a different
     slot." with a "Back to availability" CTA. The selection is
     cleared.
   - On 502 → show "Something went wrong with your payment. Our
     team has been notified and will contact you." with a
     "Contact support" link (placeholder route is fine in v1).
   - On 400 (signature reject — should be rare/impossible with
     a real Razorpay payment but possible if there's an error in
     SDK setup) → show "Payment verification failed. If you were
     charged, please contact support."
   - Other 4xx/5xx → "Something went wrong. Please try again."
     with a retry option.

5. Dismiss callback (modal closed without payment):
   - Customer cancelled the payment.
   - No backend call (the initiate already happened and created
     an unused Razorpay order; that's expected — Razorpay orders
     have a TTL and stale orders are operationally normal).
   - Return to the detail page with selection intact (let them
     try again or modify).
   - State "Payment cancelled" briefly via the error display
     pattern (toast or banner).
