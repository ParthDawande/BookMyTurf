PHASE 9-staff — Staff portal. Assigned complaints + query pool
(claim-and-resolve workflow) + notifications. Smallest role portal.

Standing rules apply. Stop and report. After acceptance, all four
role portals are complete and we move to the final integration
sweep. Reuse Phase 0 error shape, Phase 2 security UNCHANGED. The
backend is frozen for substantive business logic; small surgical
additions allowed per the established pattern if a needed endpoint
doesn't exist (verify first, decide separately).

SCOPE: three things in this sub-phase.
- StaffNav (Complaints / Queries / Notifications).
- /staff/complaints — assigned complaints list with notes + resolve.
- /staff/queries — query pool (claim) + my queries (notes + resolve).
- /staff/notifications mounts the reusable NotificationsPage with
  StaffNav prop.

Same architectural pattern as 9-owner-foundation-and-dashboard and
9-admin-foundation-and-dashboard combined into one sub-phase
because staff has less surface to cover.

=== Locked design decisions ===
- Q1: Staff lands at /staff/complaints. /staff stub from
  9-foundation becomes a redirect to /staff/complaints. No
  staff dashboard with KPIs — the assigned-complaints list IS
  the landing page.
- Q2: One page at /staff/queries with two tabs: "Pool"
  (unclaimed OPEN queries available to claim) and "Mine"
  (claimed/assigned queries). Default tab "Mine" — staff's own
  work first.
- A small "what's on my plate" banner above the complaint list
  showing assigned count + claimed query count for at-a-glance
  context.
- Notes are visible to staff and admin (internal only per 7B Q1),
  NEVER to customer. Frontend doesn't need to enforce this since
  it's a staff-only page, but it's worth being conscious that
  notes are internal.

=== Pre-build endpoint discovery (verify all via curl) ===

ASSIGNED COMPLAINTS:
- GET /api/staff/complaints (assigned to this staff). Likely accepts
  status, page, page_size. Verify shape.
- POST /api/staff/complaints/{id}/notes — add a note. Body shape?
- GET /api/staff/complaints/{id}/notes — list notes for a complaint.
- POST /api/staff/complaints/{id}/resolve — resolve with
  resolution_note (same body as admin's resolve).

QUERY POOL + MINE:
- GET /api/staff/queries — likely accepts mode ("pool" / "mine") OR
  a status filter that achieves the same. Verify how the backend
  exposes this. Per 7C the queries table has picked_up_by_staff_id;
  "pool" probably means picked_up_by_staff_id IS NULL, "mine" means
  picked_up_by_staff_id = current_user.
- POST /api/staff/queries/{id}/claim — atomic claim per 7C
  (UPDATE ... WHERE id = ? AND status = 'OPEN' with row-count check
  for race detection).
- POST /api/staff/queries/{id}/notes — add note (likely same shape
  as complaint notes).
- GET /api/staff/queries/{id}/notes — list notes.
- POST /api/staff/queries/{id}/resolve — resolve with resolution_note.

If any endpoint doesn't exist, surface it before building. Don't
assume — verify each via curl with staff9a@bmt.test's token.

=== /staff/complaints page ===

Layout:
- Header + StaffNav with Complaints active.
- Page title: "My Complaints" or "Assigned to Me."
- Banner at top: "You have {assignedCount} complaints assigned and
  {claimedCount} queries claimed." (Pull claimedCount via a separate
  fetch to /api/staff/queries?mode=mine or equivalent.)
- Status filter: All / IN_PROGRESS / RESOLVED. Default
  "IN_PROGRESS" — staff's active work first. RESOLVED tab lets
  staff see their work history.
- List of complaint cards. Each card:
  - Customer name, email, subject.
  - Description excerpt (if list response includes it; otherwise
    "View details" link to a detail page).
  - Related booking (if booking_id set, show "Re: Booking #X at
    Turf Y" text).
  - Status badge.
  - Created date.
  - Notes count (from the notes list or a count field).
  - "Add note" CTA.
  - "Resolve" CTA (conditional on status — only for IN_PROGRESS).

Add note flow:
- Modal or inline expand: textarea for note content. Submit →
  POST notes endpoint. 201 → list refetches, notes count updates.
  No toast.

Resolve flow:
- Modal: "Resolve complaint #X" with required resolution_note
  textarea (matches admin resolve flow from 9-admin-operations).
- Submit → POST /api/staff/complaints/{id}/resolve with
  {resolution_note}. 200 → list refetches, complaint now in
  RESOLVED status.
- Empty resolution_note → submit disabled.

If the list response doesn't include enough info to show the full
complaint context, add a /staff/complaints/{id} detail page that
shows the full description, customer info, all notes, and the
action buttons. Decide based on the list shape — verify via curl.

=== /staff/queries page ===

Layout:
- Header + StaffNav with Queries active.
- Page title: "Queries."
- Two tabs at the top:
  - "Mine" (default, shows assigned-to-me queries — filterable
    by IN_PROGRESS / RESOLVED, default IN_PROGRESS).
  - "Pool" (unclaimed OPEN queries available to claim).

Tab "Mine":
- Same shape as complaints. Each query card has Add Note + Resolve
  buttons (conditional on status).
- Resolve modal same shape as complaints — required resolution_note.

Tab "Pool":
- List of OPEN unclaimed queries.
- Each card: customer name, subject, description excerpt, created
  date, "Claim" button.
- Claim flow:
  - Click "Claim". No confirmation modal (claiming is reversible
    in v1 if staff just doesn't resolve it; not destructive).
  - Network: POST /api/staff/queries/{id}/claim.
  - 200 → the query disappears from Pool tab and appears under
    Mine tab.
  - 409 (race — another staff member just claimed it): error
    banner "This query was just claimed by another staff member.
    Please pick another." Refetch the pool list.
  - Other 4xx/5xx: generic error.

The atomic-claim race (7C) is real and worth verifying — two staff
members clicking claim on the same query at the same time, one
gets 200 and the other gets 409. v1 verification can simulate
this via a curl race or a controlled DB UPDATE between the GET and
the POST. State your approach.

No new code beyond the route mount + Nav prop.

=== Carried-forward contracts ===

1. The state machine: complaint must be IN_PROGRESS to be resolved.
   Since assigned complaints are always IN_PROGRESS (admin assigned
   them or staff claimed them — both transitions move status), the
   Resolve button just works without UI gating. Verify.
2. Notes are internal — staff page is staff-only, no leak by design.
3. P5: page-state-as-confirmation. No toasts.
4. §5: ZERO customer-facing notifications. Resolve fires no
   notifications per the 7B/7C contracts (resolution is internal
   tracking).
5. Snake_case throughout.
6. The reusable NotificationsPage component now mounts in four
   role portals total: customer, owner, admin, staff. State that
   StaffNav is the fourth Nav prop value.
7. Atomic claim (7C): the row-count check on UPDATE
   queries SET picked_up_by_staff_id = ? WHERE id = ? AND status = 'OPEN'
   determines success vs 409.

=== Do NOT in this sub-phase ===
- No customer/owner/admin side changes.
- No backend changes for substantive logic.
- No new dependencies.
- Don't build a separate staff dashboard with KPIs — landing is
  the complaints list.
