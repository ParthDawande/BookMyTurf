SCOPE: three notification read/update endpoints accessible by ALL
authenticated user roles. The endpoints scope by caller's user_id —
each user only ever sees their own notifications.

=== Endpoints ===

GET /api/notifications
  Authenticated, any role. Lists the caller's own notifications.
  Query parameters (all optional):
    page (default 0)
    size (default per API_DOC; if unspecified, default 20)
    is_read (optional: true/false; default omitted = all)
  Sort: most recent first (created_at DESC). State this explicitly.
  Response shape per API_DOC. Each item:
    {
      "id": <long>,
      "type": "<NotificationType enum string>",
      "message": "<string>",
      "is_read": <bool>,
      "created_at": "<ISO timestamp>"
    }
  Plus paging metadata per API_DOC (total, page, size, etc.).

PUT /api/notifications/{id}/read
  Authenticated, any role. Marks the specified notification as
  read. Idempotent: PUT-ing an already-read notification returns
  200 with the current state (NOT 400, NOT a no-op silent
  response — the response body should consistently show the
  current state).
  Authorization: caller must own the notification (user_id matches).
  Wrong-user → 404 (no-leak, NOT 403).
  Response:
    {
      "id": <id>,
      "is_read": true,
      "read_at": "<ISO timestamp of when this read occurred>" or
                 the original read timestamp if it was already read.
    }

PUT /api/notifications/mark-all-read
  Authenticated, any role. Bulk-marks all the caller's unread
  notifications as read.
  No body. Idempotent: if no unread, returns 200 with count=0.
  Response:
    {
      "marked_count": <integer — number of notifications flipped
                       from unread to read in this call>
    }
  This is a single SQL update for performance:
    UPDATE notifications SET is_read=1, read_at=NOW()
    WHERE user_id=:caller AND is_read=0
  The marked_count is the rowsAffected.

=== Schema notes ===
- The notifications table has user_id (FK), type (enum), message,
  is_read (boolean default false), created_at, and per the schema
  may or may not have read_at. State what columns exist via
  DESCRIBE notifications.
- If read_at doesn't exist, add it as a sub-§8 deviation (DATETIME(6)
  nullable). Same kind of additive timestamp as Phase 7's
  resolved_at — persistent state-transition timestamp, useful for
  the response body. State the deviation if added.
- If read_at already exists, use it. Mark-as-read writes
  read_at = NOW(); mark-all-read writes the same; unread state has
  read_at = NULL.

=== Per-user filtering (THE critical invariant) ===
- EVERY endpoint filters by user_id = caller. No cross-user leak
  is acceptable.
- The Phase 2 auth filter provides the caller's id; the service
  uses it without trust-of-input.
- GET /api/notifications: WHERE user_id=:caller.
- PUT /api/notifications/{id}/read: WHERE id=:id AND user_id=:caller
  (atomic — if rows affected = 0, return 404 because either the
  notification doesn't exist or doesn't belong to caller, indistinguishable).
- PUT /api/notifications/mark-all-read: WHERE user_id=:caller AND
  is_read=0 (rows affected = marked_count).

=== Idempotency on PUT /{id}/read ===
- Already-read notification PUT → 200 with current state.
- Never-read notification PUT → 200 with is_read=true, read_at=NOW().
- Wrong-user notification PUT → 404 (the atomic-UPDATE-by-row-count
  approach naturally produces this: rows affected = 0 → return 404).
- Race condition: two callers' simultaneous PUT on the same
  notification — handled by the atomic UPDATE. Whichever wins
  first sets read_at; the second's UPDATE affects 0 rows (because
  is_read=0 no longer matches), and we return 200 with the existing
  state (re-SELECT the row after the UPDATE). State explicitly
  how you handled this — straightforward atomic-UPDATE-then-SELECT
  is fine for v1.
