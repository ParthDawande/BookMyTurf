=== Locked design decisions (from Phase 7 framing and §8) ===
- DECISIONS §8: queries has NO picked_up_at column. The claim
  timestamp is derived server-side at the moment of claim, returned
  in the response only — NOT persisted to the DB. Do not add the
  column.
- Atomic claim: single UPDATE queries SET status='IN_PROGRESS',
  staff_id=:caller WHERE id=:id AND status='OPEN' with a row-count
  check. Row count 1 = won. Row count 0 = already claimed (or doesn't
  exist) → 409 "This query has already been claimed" (resource-
  uniqueness conflict per the 7A/7B-established 400-vs-409 rule:
  409 because the OPEN status is the "resource" that's already been
  taken).
- No SELECT FOR UPDATE. The atomic UPDATE-WHERE handles the race
  internally without explicit locking.
- query_notes XOR semantics: every row written in 7C has query_id
  SET and complaint_id NULL. The XOR is enforced application-side
  (same shared table as complaint_notes from 7B). Verify with raw
  SELECT in your report.
- Resolution requires a non-empty resolution_note (same as 7B):
  the resolve endpoint writes a query_notes row, then flips status
  to RESOLVED in the same transaction.
- Admin override-resolve: admins can resolve any query regardless
  of who claimed it. Same shape as 7B's admin override.

=== Status flow ===
OPEN → IN_PROGRESS (staff claim, atomic UPDATE)
IN_PROGRESS → RESOLVED (claiming staff or admin resolves)
No other transitions. State violations → 400 (status-transition is
state, not uniqueness).
- Claim on IN_PROGRESS or RESOLVED → 409 (uniqueness — the claim
  slot is taken or gone).
- Resolve on OPEN (unclaimed) → 400, "Query must be claimed before
  resolution."
- Resolve on RESOLVED → 400 "Query is already resolved."
- Notes on RESOLVED → 400 "Cannot add notes to a resolved query."

=== Endpoints (per API_DOC paths) ===

CUSTOMER:
- POST /api/customer/queries — create. Body: subject, description.
  Validation: non-empty.
- GET /api/customer/queries — list own queries, paginated. Status
  visible. Notes NOT included (Q1 from 7B carried — internal-only).
- GET /api/customer/queries/{id} — detail. Own only (404 otherwise).
  Notes NOT included.

STAFF:
- GET /api/staff/queries?status=OPEN — list unclaimed queries (the
  pool). Default filter: status=OPEN. Notes NOT relevant here.
- POST /api/staff/queries/{id}/claim — atomic claim. NO body.
  Response includes derived claimed_at (server timestamp at claim,
  not persisted).
- GET /api/staff/queries?status=IN_PROGRESS&mine=true — list
  queries claimed by this staff. Notes included.
- GET /api/staff/queries/{id} — detail. Only if claimed by this
  staff (404 otherwise, no-leak). Notes included.
- POST /api/staff/queries/{id}/notes — add a note. Body: note
  (non-empty). Only the claiming staff can add notes. Writes a
  query_notes row: query_id SET, complaint_id NULL, author_id=
  caller, note_text.
- POST /api/staff/queries/{id}/resolve — body: resolution_note
  (non-empty). Only the claiming staff. Writes a query_notes row
  as resolution note, then flips status to RESOLVED in same tx.

ADMIN:
- GET /api/admin/queries?status= — list all, filterable. Notes
  included.
- GET /api/admin/queries/{id} — detail with notes.
- POST /api/admin/queries/{id}/resolve — body: resolution_note.
  Admin can resolve any query regardless of claim. Same DB writes
  as staff resolve (resolution-note row + status flip), but
  author_id = the admin.
- POST /api/admin/queries/{id}/notes — optional (per 7B's
  symmetric admin-notes endpoint). State whether you built it. If
  built, admin can add notes on any query; author_id = admin.

=== JPA mapping caution (from 7A/7B) ===
Apply the same preemptive caution from 7B's complaint mapping: do
NOT set up Query with a bidirectional @OneToMany collection of
notes with orphanRemoval=true. Use a plain @ManyToOne on QueryNote
pointing to Query, with no inverse collection on Query. There is
no delete-note endpoint in 7C so the silent-rollback bug from 7A
cannot bite, but the mapping should be safe by design, not safe
because no caller exercises it.
