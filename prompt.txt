=== Locked design (from 8B-owner acceptance) ===
- Date range filter: applies to bookings.booking_date for
  booking-domain queries, payouts.scheduled_at for payout-domain
  queries. Same dual-domain rule as 8B-owner — one UI date range,
  two different SQL columns.
- Revenue formula: SUM(bookings.total_amount) for CONFIRMED +
  COMPLETED, platform-wide (no WHERE owner_id clause).
- Platform commission = SUM(bookings.commission_amount) for
  CONFIRMED + COMPLETED. THIS is what the platform actually earned.
- All sums use BigDecimal.
- Tz pattern: booking_date is LocalDate, compared directly; for
  scheduled_at use the same .atStartOfDay() Asia/Kolkata → Instant
  bounds pattern from 8A / 8B-owner.

=== Endpoint ===

GET /api/admin/dashboard
  Query parameters (all optional):
    from_date (IST date, YYYY-MM-DD)
    to_date   (IST date, YYYY-MM-DD)
    (NO turf_id, NO owner_id — admin is platform-wide. If
    API_DOC specifies admin-side drill-down filters, follow it
    and state which.)
  Response (JSON):
    {
      "total_revenue":      <SUM(bookings.total_amount) for
                            CONFIRMED+COMPLETED in date scope —
                            platform-wide gross>,
      "platform_commission":<SUM(bookings.commission_amount) for
                            CONFIRMED+COMPLETED — what the platform
                            earned>,
      "owners_payout_total":<total_revenue minus
                             platform_commission — the gross paid
                             to all owners>,
      "bookings_by_status": {
        "CONFIRMED": <count>,
        "COMPLETED": <count>,
        "CANCELLED": <count>,
        "REFUNDED":  <count>
      },
      "payouts_summary": {
        "pending_amount":   <SUM WHERE status=PENDING — date
                            filter on scheduled_at>,
        "paid_amount":      <SUM WHERE status=PAID>,
        "cancelled_amount": <SUM WHERE status=CANCELLED>
      },
      "pending_approvals": {
        "turfs":      <count of turfs WHERE status=PENDING>,
        "sub_courts": <count of sub_courts WHERE status=PENDING>
      },
      "open_complaints": <count of complaints WHERE status IN
                          ('OPEN','IN_PROGRESS')>,
      "open_queries":    <count of queries WHERE status IN
                          ('OPEN','IN_PROGRESS')>,
      "active_users": {
        "customers": <count of users WHERE role='CUSTOMER' AND
                     status='ACTIVE'>,
        "owners":    <count WHERE role='OWNER' AND status='ACTIVE'>,
        "staff":     <count WHERE role='STAFF' AND status='ACTIVE'>
      },
      "filters": {
        "from_date": "YYYY-MM-DD" or null,
        "to_date":   "YYYY-MM-DD" or null
      }
    }

  Behavior:
  - bookings_by_status: filter by booking_date range. All four
    statuses ALWAYS present, 0 if no rows match.
  - total_revenue / platform_commission / owners_payout_total:
    filter by booking_date, CONFIRMED+COMPLETED only.
  - payouts_summary: filter by scheduled_at (IST→UTC bounds, same
    as 8B-owner). All three statuses always present.
  - pending_approvals: NO date filter. Approvals queue is a snapshot
    of current state, not a historical aggregate. State this
    explicitly.
  - open_complaints / open_queries: NO date filter, same rationale —
    snapshot of current operational queue, not a historical
    aggregate.
  - active_users: NO date filter — it's a current-state snapshot.
    Count users WHERE status='ACTIVE' (the per-request DB status
    check from Phase 2 already filters out banned/suspended; we
    count the role-filtered ACTIVE subset).

=== Why the new fields have no date filter ===
- Pending approvals, open complaints/queries, active users are
  CURRENT operational snapshots. An admin asking "what's on my
  plate right now" doesn't want "things that were pending in
  May 2026" — they want the current backlog.
- Bookings/revenue/payouts are HISTORICAL aggregates by their
  nature, so the date filter makes sense there.
- State this rule explicitly in the report.

=== Admin role-scoping ===
- Admin role only. Other roles (CUSTOMER, OWNER, STAFF) → 403.
- Phase 2 security routing already enforces this for /api/admin/**.
- No 404 cases (admin sees everything; nothing to leak).

=== Carried-forward contracts applied ===
1. Race-recovery rows (booking_id IS NULL in payments/refunds): the
   dashboard never queries payments or refunds, so structurally N/A.
   State explicitly.
2. bookings.total_amount is the source of truth — DO NOT sum
   payments minus refunds.
3. opaque payment_method: irrelevant for dashboard aggregations,
   never appears.
4. Tz pattern from 6B/6C/8A/8B-owner: explicit Asia/Kolkata
   conversion for scheduled_at bounds.
5. status filters:
   - Revenue: CONFIRMED + COMPLETED.
   - Active users: status='ACTIVE'.
   - Pending approvals: turfs.status='PENDING',
     sub_courts.status='PENDING'.
   - Open operational items: complaints.status IN
     ('OPEN','IN_PROGRESS'), same for queries.

=== Active-users count: tiebreaker on the per-request DB status check ===
The Phase 2 security filter does a per-request DB status check on
the calling user. The active_users count in 8B-admin is a snapshot
of stored status; it does NOT re-validate every user's currentness.
For v1 this is fine — banned users have status='BANNED' or
'SUSPENDED' written to the DB by Phase 4's admin actions; ACTIVE
means literally what the column says. State this so future-you
doesn't worry the count drifts from "real" activity.
