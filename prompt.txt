=== SITE B — PAYMENT-race auto-refund failure FAILED record pair ===

Current state: in 6C-iii-2, on Razorpay refund failure during the
race-after-payment branch (the customer paid the additional
charge, then the slot race was detected, then the auto-refund
call failed), the catch block rethrows 502 and writes nothing.

New behavior:
- On RazorpayException + IOException in the race-refund catch:
  call the recovery service to write TWO rows via REQUIRES_NEW
  (in one method, one transaction):
    a. A payments row recording the captured additional payment:
       - booking_id = this booking (NOT NULL — Q1a still).
       - razorpay_payment_id = the additional-charge payment id
         (which was already captured before the race re-check).
       - gateway_transaction_id = same value (per existing
         payments-row convention).
       - amount = serverPriceDiff (the captured additional
         amount).
       - status = SUCCESS (the CAPTURE succeeded; only the
         refund failed).
       - payment_method = the opaque string Razorpay returned
         for that payment (fetch with payments.fetch if you
         don't already have it; or store NULL if unfetchable —
         state which).
    b. A refunds row recording the failed refund attempt:
       - booking_id = this booking (NOT NULL).
       - payment_id = the just-written payments row's id.
       - razorpay_refund_id = NULL.
       - amount = serverPriceDiff (the intended refund amount,
         which equals the captured amount).
       - status = FAILED.
  Then rethrow 502 with the standard body.
- This is the symmetric counterpart to the success path in
  6C-iii-2: both write a payments row + a refunds row;
  6C-iii-2's success writes status=SUCCESS for both, this
  sub-phase writes payments SUCCESS + refunds FAILED.

Why two rows here vs one row at Site A: at Site A there's
already a SUCCESS payment row in the DB (from the original
booking creation); the FAILED row only needs to reference it.
At Site B the additional payment has NO DB record yet (it was
captured by the customer in initiate, never confirmed because
the race caused the booking transaction to roll back); so
both a payments row recording the capture AND a refunds row
recording the failed refund attempt are needed to give the
operator queue a complete picture.

=== Operator queue semantics (extended) ===

The refunds table now has FAILED rows from four sources:
  - 5C-2b race-recovery (booking_id IS NULL) — payment_id
    references a race-recovery payments row with booking_id NULL.
  - 6B-ii cancel refund-failure (booking_id IS NOT NULL).
  - 6C-iii-3 reschedule-REFUND failure (booking_id IS NOT NULL,
    Site A).
  - 6C-iii-3 PAYMENT-race refund failure (booking_id IS NOT NULL,
    Site B).

The booking_id IS NULL vs IS NOT NULL distinction still cleanly
separates race-recovery (no real booking attached) from
booking-attached failures. Within booking_id IS NOT NULL, the
three failure types are NOT structurally distinguishable in the
refunds table alone — they're distinguishable only by which
booking-status / payments-row context they appear with. Acceptable
for v1; an operator looking at the row will look up the booking
and see what state it's in. v2 polish would be to add a
"failure_source" column or similar, but DECISIONS §8 forbids new
columns and a queue that's investigatable via JOINs is the v1
shape. State this in your report; no action required.
