=== Locked design (from Phase 8 framing) ===
- Revenue formula: SUM(bookings.total_amount) for CONFIRMED +
  COMPLETED bookings of this owner's turfs. REFUNDED and CANCELLED
  excluded. NOT SUM(payments) - SUM(refunds) — see rationale below.
- Date range filter: applies to bookings.booking_date (IST dates).
  Convert IST date range to bounds using the Phase 6B/8A pattern.
  Payouts dashboard items use payouts.scheduled_at (separate
  filter, already established in 8A).
- Owner scoping: every aggregation traverses bookings → sub_courts
  → turfs → owner_id WHERE owner_id = caller.

=== Carried-forward financial contracts to apply ===
1. Race-recovery rows have booking_id IS NULL. NOT applicable to
   bookings/payouts queries directly (those rows live in payments/
   refunds, not bookings). State this explicitly in your report.
2. bookings.total_amount is the source of truth for current state.
   Use this for revenue, not sum-of-payments.
3. payment_method is opaque varchar — irrelevant for dashboard
   aggregations.
4. Tz pattern (carried from 6B-i, 6C-ii, 6C-iii, 8A): convert IST
   dates to Instant boundaries via .atStartOfDay() in Asia/Kolkata
   for from_date and .atTime(LocalTime.MAX) for to_date. Compare
   against booking_date directly when possible (since booking_date
   is a date, not a timestamp), or convert booking_date through
   the same zone.

=== Endpoints to build ===

GET /api/owner/dashboard
  Query parameters (all optional):
    from_date (IST date, YYYY-MM-DD)
    to_date   (IST date, YYYY-MM-DD)
    turf_id   (filter to a specific turf this owner owns;
               wrong-owner → 404)
  Response (JSON):
    {
      "total_revenue": <BigDecimal — SUM(total_amount) for
                       CONFIRMED+COMPLETED in scope>,
      "total_commission": <BigDecimal — SUM(commission_amount)
                          for CONFIRMED+COMPLETED in scope; this is
                          what the platform deducts; owner sees this
                          for transparency>,
      "owner_payout_total": <BigDecimal — total_revenue minus
                            total_commission, the owner's gross>,
      "bookings_by_status": {
        "CONFIRMED": <count>,
        "COMPLETED": <count>,
        "CANCELLED": <count>,
        "REFUNDED":  <count>
      },
      "payouts_summary": {
        "pending_amount": <SUM(amount) WHERE status=PENDING>,
        "paid_amount":    <SUM(amount) WHERE status=PAID>,
        "cancelled_amount":<SUM(amount) WHERE status=CANCELLED>
      },
      "filters": {
        "from_date": "YYYY-MM-DD" or null,
        "to_date":   "YYYY-MM-DD" or null,
        "turf_id":   <id> or null
      }
    }

  Behavior:
  - bookings_by_status counts: filter by booking_date range if
    given. All four statuses ALWAYS appear in the response, with
    0 if no bookings match.
  - revenue/commission/payout: filter by booking_date range,
    CONFIRMED+COMPLETED only.
  - payouts_summary: filter by payouts.scheduled_at range (NOT
    booking_date — payouts have their own date dimension). Show
    all three statuses always, zero if no rows.
  - All sums use BigDecimal (HALF_UP for any rounding that occurs
    on the SUM, though SUMs of stored DECIMAL(10,2) shouldn't need
    rounding).
