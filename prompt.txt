- P4: review CTA on receipt visible only when status=COMPLETED.

=== Backend endpoints used (existing, do not modify) ===
- POST /api/customer/bookings/{id}/reschedule/initiate — body:
  new booking_date and new slots (state the exact shape via curl
  before building UI around it). Response: action_required (NONE/
  REFUND/PAYMENT), price_diff, refund_amount or razorpay order

- PUT /api/customer/bookings/{id}/reschedule/confirm — body: the
  reschedule_token; for PAYMENT case also the three Razorpay
  values. Response: updated booking detail.
- POST /api/customer/reviews — body: booking_id, rating, comment.
- PUT /api/customer/reviews/{id} — body: rating, comment.
- DELETE /api/customer/reviews/{id}.

=== Reschedule page (/customer/bookings/:id/reschedule) ===

Single-page layout:

LEFT/TOP — "Current booking" panel (read-only summary):
- Turf, sub-court, current date, current slots, current total.
- "Cancel reschedule" link → back to receipt unchanged.

RIGHT/BOTTOM — "New booking" panel (interactive):
- Date picker. Default = current booking_date. Selecting a new
  date triggers an availability fetch for that date.
- Availability grid for the SAME sub-court (cross-sub-court
  reschedule is NOT supported per the 6C Q1 decision). Use the
  same multi-slot selection from 9-customer-booking: free/taken/
  selected visual states, contiguous-only enforcement (choice a).
- New total calculated client-side: selected slots count × rate.
- Price-difference summary, live-updated as selection changes:
  - "No price change" if diff == 0.
  - "You'll be refunded ₹X" if diff < 0.
  - "Additional ₹X to pay" if diff > 0.
- "Continue" CTA, disabled if no slots selected or selection is
  the same as the current booking (no-op reschedule).

The "Continue" CTA fires the initiate call. From the response's
`action_required` field, the next step diverges:

CASE NONE (equal price):
- Confirmation modal: "Reschedule confirmed?" with old vs new
  summary and a "Confirm" CTA.
- Click → PUT /reschedule/confirm with the token. No Razorpay.
- 200 → navigate to /customer/bookings/{id}. The receipt
  reflects the new slots/date.
- Errors per the mapping below.

CASE REFUND (price decrease):
- Confirmation modal: "Confirm reschedule? You'll be refunded
  ₹X.XX." with old vs new + the refund amount.
- Click → PUT /reschedule/confirm with the token. No Razorpay.
- 200 → backend issues the partial refund, returns updated
  booking. Navigate to receipt → shows new slots/date + the
  partial refund visible in the refunds array.
- Errors per the mapping.

CASE PAYMENT (price increase):
- Review modal: "Pay ₹X additional to reschedule?" with old vs
  new + the additional charge. CTA: "Pay ₹X.XX now."
- Click → open Razorpay checkout.js with the initiate
  response's order_id, key_id, amount. Same SDK loader pattern
  from 9-customer-booking.
- Razorpay success callback → PUT /reschedule/confirm with the
  token AND the three Razorpay values.
- 200 → navigate to receipt. Now shows: new slots/date, total
  reflecting the increase, payments array with BOTH the
  original and the additional charge (the multi-payment audit
  trail that lifecycle-1 already renders).
- Razorpay dismiss → "Payment cancelled" banner, selection
  intact, no confirm call fired.
- Errors per the mapping (race-after-payment 409, etc.).

=== Error mapping for reschedule (state the table in your report) ===
| Backend response on confirm | Frontend behavior |
|---|---|
| 200 OK | Navigate to /customer/bookings/{id} (receipt updated) |
| 400 invalid token (expired or tampered) | "Reschedule session expired. Please start again." → back to receipt |
| 409 pricing-change | "Rates changed since you started. Please review and try again." → back to /reschedule with same date selected (refetch availability/pricing) |
| 409 slot-race | "One of your selected slots was just taken. Please pick again." → stay on /reschedule, refetch availability for the date |
| 502 refund failure (REFUND case) | "Reschedule failed. Your booking is unchanged. Please try again." (same shape as cancel-502 from lifecycle-1) |
| Other 4xx/5xx | "Something went wrong. Please try again." |

For the PAYMENT case specifically, race-after-payment is the 409 the customer needs to understand: "This slot was taken after you paid. Your additional payment has been refunded — please pick a different slot." Same shape as 9-customer-booking's 409.

=== Reviews on COMPLETED bookings ===

On the receipt page (lifecycle-1), the "Leave a review" placeholder
button only appears for COMPLETED bookings. This sub-phase
implements the actual form.

Review widget (modal or inline expandable section):
- Star rating: 5-star selector (click to set, hover to preview).
- Comment textarea (note: backend may require non-empty — let the
  backend's validation drive; show backend error if so).
- Submit → POST /api/customer/reviews. On 200, the receipt
  refetches and shows the review attached.

If a review exists for the booking:
- The button on the receipt changes to "View/edit your review."
- Clicking opens the same widget pre-filled with the existing
  rating + comment.
- Submit → PUT /api/customer/reviews/{id}.
- Delete button (with confirmation modal: "Delete your review?
  This cannot be undone.") → DELETE → on success, refetch and
  the button reverts to "Leave a review."

The review shows on the receipt page in a small "Your review"
section, displaying the rating, comment, posted-on date. If an
owner reply exists for this review (from the owner-reply feature
in 7A), show it below the review with "Owner reply:" prefix.

=== Carried-forward contracts to enforce ===

1. Razorpay SDK reload: the loader from 9-customer-booking
   (utils/razorpay.js) is reused; the SDK loads once per session
   and is reused for both bookings and reschedule-PAYMENT.
2. Contiguous-only slot selection (lifecycle-1's pattern carries
   over).
3. Multi-payment receipt rendering from lifecycle-1 — the
   PAYMENT reschedule will create a booking with 2 payments rows,
   and the receipt will render both per the 6A receipt fix.
4. Backend's reschedule_token has a 15-minute expiry (6C-i).
   The frontend shouldn't try to refresh or extend it; if it
   expires, the customer starts over.
5. No customer-facing notifications. P5 page-update-as-confirmation.

=== Do NOT in this sub-phase ===
- No cross-sub-court reschedule (locked at 6C Q1).
- No reschedule on COMPLETED, CANCELLED, REFUNDED bookings.
- No "reschedule history" view (out of scope — schema
  doesn't retain slot history per the 6C Q3 decision).
- No backend changes.
