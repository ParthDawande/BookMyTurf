PHASE 9-customer-other — Customer complaints, queries, and
notifications inbox. The closing piece of the customer side.

Standing rules apply. Stop and report. After acceptance, the
customer side is fully done and we move to 9-owner.
Reuse Phase 0 error shape, Phase 2 security UNCHANGED. The backend
is frozen — build against existing endpoints. No new dependencies.

SCOPE: three customer-facing pieces.
- Raise a complaint: from a booking receipt OR standalone form.
- Raise a query: same shape as complaints.
- Notifications inbox: /customer/notifications. Lists the
  customer's notifications (will be EMPTY for customers per §5)
  with mark-read and mark-all-read UI. This component is built
  so it can be reused by /owner/notifications, /admin/notifications,
  /staff/notifications later — but for now it only renders for
  the customer portal.

=== Backend endpoints used (existing, do not modify) ===
- POST /api/customer/complaints — create complaint.
- GET /api/customer/complaints — list customer's own complaints.
- GET /api/customer/complaints/{id} — detail (no notes for customer
  per 7B Q1 — internal-only).
- POST /api/customer/queries — create query.
- GET /api/customer/queries — list customer's own queries.
- GET /api/customer/queries/{id} — detail.
- GET /api/notifications — paginated, role-agnostic per 8C.
- PUT /api/notifications/{id}/read — idempotent.
- PUT /api/notifications/mark-all-read — bulk.

Verify shapes via curl before building if uncertain; state any
divergence from what we discussed in framing.

=== Pages to build ===

PAGE 1: Receipt page inline "Raise a complaint" CTA
- On /customer/bookings/{id}, add a "Report an issue with this
  booking" link or button in a subtle location (footer or
  secondary action area — not competing with primary actions
  like Cancel/Reschedule/Review).
- Visible for ALL statuses (CONFIRMED, COMPLETED, CANCELLED,
  REFUNDED) — customers may legitimately want to complain about
  any of them.
- Click opens a modal OR navigates to /customer/complaints/new
  with the booking_id pre-populated. Modal is simpler; recommend
  modal.
- Same "Have a question about this booking?" CTA for queries,
  same pattern.

PAGE 2: /customer/complaints/new — standalone complaint form
- Form fields per the 7B backend contract: subject (required),
  description (required), booking_id (optional dropdown of
  customer's CONFIRMED/COMPLETED/CANCELLED/REFUNDED bookings
  — exclude OPEN/PENDING since they're not relevant for
  complaints).
- Submit → POST /api/customer/complaints. On 201 → navigate to
  /customer/complaints/{id} (or back to a list with success
  banner — pick one and state).
- Errors: backend validation errors surfaced via the existing
  pattern.

PAGE 3: /customer/queries/new — standalone query form
- Same shape as complaints/new. Subject, description, optional
  booking_id dropdown.
- Submit → POST /api/customer/queries. On 201 → navigate to
  /customer/queries/{id}.

PAGE 4: /customer/complaints — list of customer's complaints
- Paginated. Status filter (All, OPEN, IN_PROGRESS, RESOLVED).
- Card per complaint: subject, status badge, created date,
  related booking (if booking_id is set, show "Re: Booking #X
  at Turf Y"; if null, show "General complaint").
- Empty state: "You haven't raised any complaints. If something's
  wrong, you can report it from your booking receipt or
  [raise a general complaint]."
- "Raise a complaint" CTA somewhere visible → /customer/
  complaints/new.

PAGE 5: /customer/complaints/{id} — complaint detail
- Subject, description, status, created date.
- Related booking link if booking_id set.
- NO notes shown (7B Q1 — internal-only).
- A small "Status: OPEN — we'll respond soon" or similar
  informational text per status. For RESOLVED status: "Status:
  RESOLVED on [date]" using the resolved_at column from
  Phase 7C-accepted schema deviation.
- 404 (cross-customer) → "Complaint not found" page.

PAGE 6: /customer/queries — list of customer's queries
- Same shape as complaints list. Status filter (All, OPEN,
  IN_PROGRESS, RESOLVED).

PAGE 7: /customer/queries/{id} — query detail
- Same shape as complaint detail. No notes shown. resolved_at
  for RESOLVED status.

PAGE 8: /customer/notifications — notifications inbox
- Paginated list of the calling user's notifications.
- Each item: notification type (with friendly label — e.g.,
  "BOOKING_CANCELLED" → "Your booking was cancelled" or
  whatever maps appropriately — though note that for CUSTOMERS
  per §5, no notifications ever get created, so this list is
  always empty for customers; the component is built ready for
  population when owner/admin/staff use it later).
- For empty case (the expected customer state): clean empty-
  state UI: "No notifications yet."
- For populated case (forward-prepared for other roles):
  - Each item shows type, message, created_at, is_read state
    (visual difference between read and unread — bold unread,
    normal weight read).
  - Click an unread notification → PUT
    /api/notifications/{id}/read → state updates immediately.
  - "Mark all as read" button visible only if there are unread
    items → PUT /api/notifications/mark-all-read → state
    updates immediately.
- Filterable by is_read (All / Unread / Read) per 8C's
  is_read filter support.

=== Type-label mapping for notifications ===
For each NotificationType enum value the customer might see
(in this sub-phase: probably never, but build it correctly for
forward use), provide a friendly label. Example mapping:
- BOOKING_CANCELLED → "Booking cancelled"
- BOOKING_RESCHEDULED → "Booking rescheduled"
- TURF_APPROVED → "Turf approved"
- COMPLAINT_ASSIGNED → "Complaint assigned to you"
- PAYOUT_RELEASED → "Payout released"
- etc.

State your label mapping. For any type not in the mapping,
default to the raw type string (defensive — frontend doesn't
break on unknown types).

=== Carried-forward contracts to enforce ===

1. Customer never sees the notes array on complaints/queries
   (7B Q1 — internal-only). Verify by inspecting the response
   for the customer detail endpoint and confirming no notes
   array is shown.
2. resolved_at column from Phase 7B/7C schema deviation is
   surfaced in the detail page when status=RESOLVED.
3. The customer-facing notifications list is structurally empty
   per §5 — verify this by checking the response.
4. The notifications component is reusable; later sub-phases
   (9-owner, 9-admin, 9-staff) will mount it at their own
   /<role>/notifications routes and it'll display the populated
   data correctly.

=== Do NOT in this sub-phase ===
- No edit or delete on complaints/queries (the spec doesn't
  permit it — customers create-only).
- No notes display on customer-side detail pages.
- No staff/admin views of complaints/queries (those are 9-staff
  and 9-admin).
- No backend changes.
