CONTEXT: Phase 9 frontend work surfaced that /api/public/turfs (list)
is public, but turf DETAIL and AVAILABILITY only exist under
/api/customer/** behind hasRole("CUSTOMER"). For a browse-before-
signup marketplace funnel, detail and availability should be public
(anonymous visitors evaluate a turf before creating an account; the
auth wall belongs at booking, not browsing). This sub-phase adds
public variants.

SCOPE: two new PUBLIC GET endpoints that expose EXISTING turf-detail
and availability logic without the auth gate. NO new business logic —
delegate to the same service methods (or thin public wrappers) that
the authenticated /api/customer versions already use.

=== Endpoints to add ===
- GET /api/public/turfs/{id}
  Public (no auth). Returns the same turf-detail shape as the
  existing /api/customer/turfs/{id} — turf info, sub-courts with
  photos/sport/hourly_rate/operating-hours, reviews + replies,
  cover_photo_url (always present, null-safe per 6A), avg_rating,
  review_count.
- GET /api/public/turfs/{id}/availability?date=YYYY-MM-DD
  Public (no auth). Returns the same availability shape as the
  existing /api/customer/turfs/{id}/availability — per-sub-court
  slots with start_time/end_time and the taken/free indicator.

=== How to implement (reuse, don't rewrite) ===
- The customer-authenticated versions already compute detail and
  availability. Identify those service methods.
- The public endpoints call the SAME service methods. If those
  methods take a customer/user param that they don't actually need
  for the computation, refactor so the public path can call them
  without a user. If they genuinely need a user param for some
  customer-specific enrichment, identify what that enrichment is
  and EXCLUDE it from the public response (see leak check below).
- Add the two routes to a public controller (or the existing public
  turfs controller from 5A). Update SecurityConfig to permit
  /api/public/turfs/{id} and /api/public/turfs/{id}/availability
  without authentication — these should match the same public
  pattern as the existing /api/public/turfs list.

=== Critical: only APPROVED turfs are public ===
- The list endpoint (/api/public/turfs) already filters to
  APPROVED turfs only (pending/rejected turfs aren't shown to the
  public). The public DETAIL endpoint MUST apply the same filter:
  requesting /api/public/turfs/{id} for a turf that is PENDING or
  REJECTED must return 404 (not 403, not the detail). A
  not-yet-approved turf must not be publicly viewable via the
  detail endpoint just because someone guesses its id.
- State explicitly how you enforced the APPROVED-only filter on
  the public detail and availability endpoints.

=== Critical: no customer-data leak in availability ===
- The availability response says a slot is TAKEN or FREE. It MUST
  NOT include WHO booked a taken slot — no customer name, no
  booking id, no customer id, nothing customer-identifying. A
  taken slot is just "taken."
- Verify the existing availability response shape doesn't already
  leak this; if it does (even in the authenticated version), the
  PUBLIC version must strip it. State what the availability
  response contains field-by-field and confirm no customer
  identity leaks.

=== Same-shape contract ===
- The public detail response shape should be IDENTICAL to the
  authenticated customer detail response shape, so the frontend
  uses one rendering path. If they must differ (e.g., the
  authenticated version includes a "can_review" flag or
  "your_existing_review" that's user-specific), document the
  difference and ensure the public shape is a clean subset.
- State whether the shapes are identical or where they diverge.

=== Do NOT in 8D ===
- No new business logic — reuse existing detail/availability
  computation.
- Do not change the authenticated /api/customer/turfs/{id} or its
  availability endpoint — those stay as-is for logged-in users.
- Do not weaken security on any other route. Only the two new
  public turf routes become permitAll.
- Do not expose pending/rejected turfs publicly.
- Do not leak customer identity in availability.
- No frontend changes in this sub-phase (the frontend update to
  use these public endpoints happens when we return to
  9-customer-auth-and-browse acceptance).
