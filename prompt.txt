begins. No feature pages yet beyond a basic landing.

Standing rules apply (the build-wide rules from DECISIONS.md etc.
apply to backend integration only; frontend has its own concerns).
Stop and report at the end. Do NOT start 9-customer-auth-and-browse.

SCOPE: set up the frontend foundation under ./frontend so all later
sub-phases can build feature pages against it. Routing skeleton,
axios client, auth context, error display, role guards, design
tokens, basic landing. No real feature pages.

=== Locked conventions (from build state) ===
- Frontend stack: Vite + React + React Router + Axios (already
  scaffolded in Phase 0).
- Snake_case URL params and snake_case request/response bodies
  end-to-end. Do NOT use a camelCase transformer. TypeScript types
  (if using TS) are snake_case to match the wire format. If using
  JS, just match the wire format directly.
- Backend base URL: configurable via environment variable
  (VITE_API_BASE_URL or similar), default http://localhost:8080.
  State the env var name you used.
- JWT bearer token stored in localStorage for v1. State explicitly
  this is acceptable because Phase 2's per-request DB status check
  invalidates banned/suspended users on the next request regardless
  of token staleness.

=== Project structure ===
Under ./frontend/src:
- pages/ (route-targeted components: pages/Landing.jsx, pages/Login.jsx, etc.)
- components/ (reusable presentational components: components/Button.jsx, components/Input.jsx)
- hooks/ (custom hooks: hooks/useAuth.js, hooks/useApiError.js)
- api/ (the axios instance and per-domain wrappers: api/client.js, api/auth.js)
- contexts/ (React Context providers: contexts/AuthContext.jsx)
- lib/ (utilities: lib/tokens.js for localStorage helpers)
- styles/ (design tokens, global CSS)
- App.jsx (router + provider setup)
- main.jsx (entry point)

State the actual structure you chose and any deviations.

=== Routing skeleton ===
Set up React Router routes for the four role-portals. Most are
stub pages in 9-foundation (just render the page title and "not
yet built"):
- /                       → Landing
- /login                  → Login
- /register               → Register (customer registration only;
                            owners register a different way later)
- /customer/*             → Customer routes (stub for now,
                            requires CUSTOMER role)
- /owner/*                → Owner routes (stub, requires OWNER)
- /admin/*                → Admin routes (stub, requires ADMIN)
- /staff/*                → Staff routes (stub, requires STAFF)

Use React Router's nested routing pattern. Each role-prefixed
route group wraps in a RoleGuard component that checks
authContext.user.role against the required role. If unauthenticated
→ redirect to /login. If wrong role → redirect to / with an error
message (or a simple 403 page).

=== Axios client (api/client.js) ===
A single configured axios instance:
- baseURL from env var.
- Request interceptor: read JWT from localStorage; if present, add
  Authorization: Bearer <token> header. Do NOT add the header if no
  token exists.
- Response interceptor: pass-through on success. On 401, clear the
  stored token + auth context and redirect to /login (the user's
  session is invalid).
- Other errors (400/403/404/409/500/502): pass through; let the
  calling code handle the error shape. Standard backend error
  shape is {error: "..."}.

State the interceptor logic in the report.

=== Auth context (contexts/AuthContext.jsx) ===
A React Context provider that exposes:
- user: {id, email, role, ...} or null
- token: string or null
- login(email, password): calls POST /api/auth/login, stores token
  + user in state and localStorage, returns success/error.
- logout(): clears token + user from state and localStorage.
- register(payload): calls POST /api/auth/register for a CUSTOMER
  account.
- isAuthenticated, hasRole(role) convenience functions.

On app mount, the provider hydrates user + token from localStorage
if present. State what your hydration approach is (synchronous on
mount? async with a loading flag?).

=== Error display pattern ===
Pick ONE of:
(a) A useApiError hook that returns {error, setError, clearError},
    used in each page with a shared <ErrorBanner> component.
(b) A toast/notification system using a library or custom
    component.
(c) An ErrorBoundary at the route level.

State which you picked and why. (a) is simplest for v1; (b) is most
user-friendly but more code; (c) is for unexpected errors but
doesn't help with normal API error display.

The error display must surface the backend's {error: "..."} shape.
For validation errors {error: "Validation failed", details: {...}},
display the details map cleanly (field-by-field if appropriate).

=== Role-based route guard ===
A RoleGuard component:
- Takes a role prop (CUSTOMER, OWNER, ADMIN, STAFF) or array of
  allowed roles.
- Reads auth context.
- If not authenticated → <Navigate to="/login" replace />.
- If wrong role → <Navigate to="/" replace />. Show a brief toast
  or banner explaining the redirect (optional).
- If correct role → render children.

=== Minimal design tokens ===
A CSS file or styled-components theme defining:
- Colors: primary, secondary, neutral palette, error red, success
  green.
- Typography: font family, base size, heading sizes.
- Spacing scale: 4px / 8px / 16px / 24px / 32px / 48px.
- Border radius, shadow.

Doesn't need to look polished — needs to be consistent so feature
sub-phases use the tokens, not arbitrary values.

State whether you used a CSS Module, plain CSS with variables,
styled-components, Tailwind, or similar. v1 recommendation: plain
CSS with CSS variables + minimal global stylesheet. No CSS
framework dependency.
