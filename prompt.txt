=== Locked design (from Phase 8 framing) ===
- Q1 = option (c): scheduler + admin trigger, scheduler disabled in v1.
- New config property: app.payout.scheduler.enabled (default false in
  tracked application.properties — explicitly literal false, not
  reliant on @Value default).
- New config property: app.payout.scheduler.fixed-rate-ms (the period
  between runs when enabled; default 60000 = 1 minute, irrelevant
  in v1 because scheduler is off, but settable for v2).
- Existing config app.payout.hold-hours (=24) is reused as-is.
- Service method: PayoutReleaseService.releaseEligiblePayouts() —
  the unit of work, callable from both the @Scheduled wrapper and
  the admin endpoint.
- SELECT FOR UPDATE on each candidate payout row when flipping (to
  serialize against a concurrent cancel that's also touching the
  same row). Same pattern as 5B's sub-court lock.

=== The release logic ===
1. Find all payouts WHERE status = 'PENDING'
   AND scheduled_at <= NOW().
2. For each, in its own @Transactional method:
   - SELECT ... FOR UPDATE on the payouts row.
   - Re-read its status under the lock — if not PENDING anymore
     (race against cancel), skip.
   - Re-read scheduled_at under the lock — if scheduled_at > NOW()
     (defensive), skip.
   - Set status = 'PAID', set paid_at = NOW().
   - Emit OWNER notification: PAYOUT_RELEASED (state the exact type
     used and message format).
   - Commit.
3. Return count of payouts processed.

=== Timezone handling (carried-forward Phase 6B/6C pattern) ===
- scheduled_at is stored as Instant (UTC).
- "NOW" for comparison is Instant.now() (UTC).
- DO NOT compare LocalDateTime to LocalDateTime relying on the JVM
  default zone.
- Quote the exact comparison line in your report.

=== Schema column: paid_at ===
- The payouts table has a paid_at column per the original schema
  (or, if not, it needs adding — state which). On release, set
  paid_at = Instant.now() (UTC). When status = CANCELLED (from 6B
  cancel logic), paid_at stays NULL.
- If paid_at doesn't exist in the schema, ADD it as a deviation
  (DATETIME(6) nullable). Same kind of deviation as the
  Phase 7 resolved_at columns — persistent state-transition
  timestamp for dashboard use.

=== Admin trigger endpoint ===
- POST /api/admin/payouts/run-release (per API_DOC; if not specified,
  use this path).
- No body. Returns the count of payouts released and the list of
  released payout IDs.
- Admin role only. Other roles → 403.
- Calls the same PayoutReleaseService.releaseEligiblePayouts().

=== Owner-facing payouts list ===
- GET /api/owner/payouts — list payouts for the owner's turfs.
  Paginated per API_DOC.
- Filterable by status (PENDING, PAID, CANCELLED — or none for all).
- Filterable by date range (from_date, to_date as IST date strings).
  Date range filter applies to scheduled_at: convert IST date to
  Instant bounds (atStartOfDay / endOfDay 23:59:59.999999999 in
  Asia/Kolkata) and use those as the SQL bounds. Same tz pattern as
  6B/6C-iii.
- Response includes: payout id, booking id (with maybe basic booking
  info like turf name + booking_date), amount, status, scheduled_at,
  paid_at (if PAID), created_at.

=== The carried-forward filter on payouts ===
- payouts always have booking_id NOT NULL by construction (every
  payout is for a specific booking). There's no "race-recovery"
  payout shape. The booking_id IS NULL filter from earlier phases
  applies to payments/refunds, not payouts. State this explicitly.
- A payout for a booking that has REFUNDED or has its booking_id
  in a CANCELLED state may itself be CANCELLED (from 6B cancel
  logic). Those should show in the owner's list with status =
  CANCELLED.

=== Owner scoping (no cross-owner leak) ===
- An owner viewing /api/owner/payouts sees ONLY their own payouts
  (via the booking → sub-court → turf → owner chain).
- An owner viewing /api/owner/payouts/{id} where the payout belongs
  to another owner → 404 (no-leak, same as Phase 3 turf access).
