PHASE 9-owner-turf-management — Turf CRUD + sub-court CRUD + photos
for the owner portal.

Standing rules apply. Stop and report. Do NOT start
9-owner-finance-and-engagement. Reuse Phase 0 error shape, Phase 2
security UNCHANGED. The backend is frozen — build against existing
endpoints. No new dependencies unless essential.

SCOPE: the owner manages their turfs and sub-courts. Largest single
sub-phase of 9-owner because of the breadth of CRUD operations.
Five features land here:
- Turf list (/owner/turfs) — filterable by status.
- Create turf — form, submit → PENDING.
- Turf detail with edit (/owner/turfs/:id) — sub-courts inline.
- Sub-court CRUD inline on the turf detail page.
- Photos: upload/delete on both turfs and sub-courts.

=== Locked design decisions ===
- Q1 (c): turf list filterable by status, default "all". Same shape
  as the customer's bookings list. PENDING/APPROVED/REJECTED/SUSPENDED
  (the exact set depends on the Phase 4 enum — state which statuses
  apply).
- Q2 (a): sub-courts are an inline section on the turf detail page,
  not a separate /sub-courts route.
- Q3 (edit immutability): verify via curl which fields are mutable
  on PENDING vs APPROVED turfs. Phase 3 had rules. State your
  observations.

=== Backend endpoints used (existing, do not modify) ===
Verify each shape via curl before building UI. State the actual shapes.

Turf:
- GET /api/owner/turfs (list owner's turfs, may take ?status= filter)
- POST /api/owner/turfs (create turf, → PENDING)
- GET /api/owner/turfs/{id} (detail)
- PUT /api/owner/turfs/{id} (edit, mutability rules per Phase 3)
- DELETE /api/owner/turfs/{id} (if it exists — state whether)

Sub-courts (nested under turf or top-level — verify):
- GET sub-courts for a turf
- POST create sub-court (→ PENDING)
- PUT edit sub-court
- DELETE sub-court

Photos:
- POST upload photo (state the upload mechanism: multipart form data?
  presigned URL? raw bytes? Phase 3 wired this but the protocol
  wasn't deeply verified in this conversation. Discover via curl
  AND by inspecting the controller. State what you found.)
- DELETE photo (state path).

=== /owner/turfs — turf list page ===

Layout:
- Header + OwnerNav. "My Turfs" nav item now active.
- Page title: "My Turfs."
- Status filter pills/dropdown: All / Pending / Approved / Rejected
  / (whatever statuses exist). Default "All".
- "+ Add a turf" CTA (prominent) → /owner/turfs/new.
- Turf grid or list: card per turf with:
  - Cover photo (or SVG placeholder if null — reuse the
    null-safe-cover pattern from 9-customer-auth-and-browse).
  - Turf name, city.
  - Status badge (color-coded per status).
  - Sub-court count.
  - "Edit" CTA → /owner/turfs/{id}.
- Status badges with the same visual treatment as customer bookings'
  status badges (use the design tokens for consistency).
- Empty state: "You haven't added any turfs yet. + Add your first
  turf."

URL semantics:
- /owner/turfs?status=PENDING — bookmarkable.
- Status="All" doesn't include the param.

=== /owner/turfs/new — create turf form ===

Form fields per the backend's CreateTurfRequest DTO (verify via
curl). Likely:
- Name (required)
- City (required)
- Address (required)
- Description (optional)
- Sports list (one or more — likely multi-select)
- Operating hours (per Phase 3 — if applicable)
- Any other fields the DTO has — list whatever the backend exposes.

Submit → POST. On 201 → navigate to /owner/turfs/{newId} with

The new turf starts with status=PENDING, no photos, no sub-courts.

=== /owner/turfs/:id — turf detail with edit + sub-court management ===

Layout:
- Header + OwnerNav.
- Breadcrumb back to /owner/turfs.
- Turf header section:
  - Cover photo (if present) or placeholder + "Upload cover photo"
    button.
  - Status badge.
  - Editable fields per Q3's mutability rules:
    - If PENDING: name, address, description, etc. — all editable.
    - If APPROVED: state which fields can still be edited (likely
      description, hours, etc. — NOT name/city/address). The fields
      that ARE mutable get edit-pencil affordances; the ones that
      AREN'T mutable are displayed as read-only with a small "this
      cannot be edited after approval" tooltip.
    - If REJECTED: state behavior. Likely fully editable (resubmit).
    - State your rule discovery.
  - Save changes button → PUT /api/owner/turfs/{id}.

- Photos section:
  - Existing photos rendered as a gallery.
  - "Upload photo" button → file input → POST upload endpoint.
    State the upload mechanism (multipart? base64? presigned URL?
    use whatever Phase 3 wired).
  - Each photo has a delete button → DELETE endpoint with
    confirmation modal.
  - The first photo (lowest id) is the cover photo per the 6A
    carried-forward rule — state if there's a way for the owner
    to designate a different cover (probably out of scope for v1
    — first-by-id is the rule).

- Sub-courts section:
  - List of existing sub-courts, each as a card:
    - Sub-court name, sport, hourly rate, operating hours, status
      badge, sub-court photos thumbnail row.
    - "Edit" button → opens inline edit modal OR navigates to a
      sub-court detail (state your approach).
    - "Manage photos" → opens a photos modal for that sub-court.
    - "Delete" button → confirmation modal → DELETE.
  - "+ Add a sub-court" CTA at the bottom of the list.
  - Add and edit modals share the same form per the
    CreateSubCourtRequest / UpdateSubCourtRequest DTOs. Verify
    DTOs via curl.

States:
- Loading: skeleton for the turf info, sub-court grid, photo
  gallery.
- Error: existing ErrorBanner.
- Editing in progress: optimistic UI is NOT applied; wait for the
  PUT response before showing "Saved" feedback. P5 page-state-as-
  confirmation carries: a small inline "Saved" indicator that
  fades, or the page just reflects the new values without a toast.
- 404: "Turf not found" (cross-owner attempt → 404, no-leak per
  Phase 3).

=== Photos upload mechanism ===

The upload protocol wasn't extensively verified in earlier phases
of this conversation. Investigate before building:
1. Check the controller signature for the upload endpoint.
   Multipart-form-data (Spring's MultipartFile parameter)? JSON
   with base64-encoded image? Presigned URL pattern?
2. Verify with a curl test before building the UI.
3. State the protocol explicitly.
4. Build the UI to match: an <input type="file"> for multipart;
   FileReader + base64 for JSON; etc.

If the protocol requires significant new client code (e.g., chunked
uploads, presigned URLs with separate finalize calls), note the
complexity and propose: implement the simple cases first, defer
edge cases (very large files, multiple simultaneous uploads) to
v2 polish.

Image rendering:
- After upload, the response should include a URL or id that the
  frontend uses to render the new photo. State the response shape.
- If the backend serves photo bytes directly (e.g., /api/photos/{id})
  vs. returning an external URL (e.g., S3): the rendering approach
  differs. State which.

=== Edit immutability (Q3) ===

For PENDING turfs: presumably all fields mutable.
For APPROVED turfs: per Phase 3 contracts (which we wrote in this
conversation but haven't recently re-read), some fields locked.
Likely-immutable: turf_name, city, address (because they affect
customer search and approval evaluation).
Likely-mutable: description, operating hours, additional photos.

Verify via curl: PUT an APPROVED turf with a name change. Does the
backend accept it (return 200 with the new name) or reject (return
400 / ignore the field)? Report what you observe and build the UI
accordingly. The UI should disable the immutable fields with
explanation text.

=== Carried-forward contracts ===

1. P5: page-state-update IS the confirmation. After save: page
   shows new state, no toast on top. Same for create/delete.
2. cover_photo_url null-safe rendering on the turf list and detail
   header (from 9-customer-auth-and-browse).
3. Status badges color-coded per status using the design tokens
   (consistent with customer-side status badges).
4. 404-not-403 no-leak on cross-owner attempts. Phase 3's
   structural rule carries through to the owner UI: owner A
   cannot fetch owner B's turf details — backend returns 404,
   frontend renders "Turf not found."
5. Snake_case throughout.
